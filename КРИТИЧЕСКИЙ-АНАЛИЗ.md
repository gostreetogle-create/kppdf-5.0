<a id="heading-3"></a>
## 1. КРИТИЧЕСКИЕ ПРОБЛЕМЫ (P0) — ИСПРАВИТЬ НЕМЕДЛЕННО

<a id="heading-4"></a>
### 1.1. JWT secret захардкожен
- **Файл:** `src/lib/auth.ts:5`
- **Проблема:** `const JWT_SECRET = process.env.JWT_SECRET || 'kppdf-dev-secret-key-change-in-production'`
- **Риск:** Если `.env` не содержит `JWT_SECRET`, используется предсказуемый секрет. Любой может подделать JWT.
- **Исправление:** Требовать `JWT_SECRET` в env, падать при его отсутствии.

<a id="heading-5"></a>
### 1.2. Seed endpoint без авторизации
- **Файл:** `src/app/api/seed/route.ts:5`
- **Проблема:** `POST /api/seed` не вызывает `requireAuth()`
- **Риск:** Любой может перезаписать все тестовые данные в продакшене.
- **Исправление:** Добавить `requireAuth()` + `requireRole(['admin'])`.

<a id="heading-6"></a>
### 1.3. Upload без валидации типа файла
- **Файл:** `src/app/api/upload/route.ts:33`
- **Проблема:** Расширение берётся из имени файла клиента, нет проверки MIME-type
- **Риск:** Можно загрузить `.exe`, `.php`, `.js` файл. Файлы доступны напрямую через `/uploads/`.
- **Исправление:** Whitelist допустимых расширений + проверка MIME + limitations на размер.

<a id="heading-7"></a>
### 1.4. Нет валидации входных данных
- **Файлы:** Все API routes
- **Проблема:** `zod` установлен, но нигде не используется. Тело запроса напрямую передаётся в Prisma.
- **Риск:** Любой клиент может отправить произвольные поля и они запишутся в БД.
- **Исправление:** Добавить Zod-схемы для каждого API endpoint.

<a id="heading-8"></a>
### 1.5. Нет role-based access control
- **Файлы:** Большинство API routes
- **Проблема:** DELETE proposals, contracts, production-orders выполняется любым авторизованным пользователем (включая viewer).
- **Риск:** Viewer может удалить производственный заказ.
- **Исправление:** `requireRole(['admin', 'manager'])` для DELETE/PUT операций.

<a id="heading-9"></a>
## 2. ВЫСОКИЕ ПРОБЛЕМЫ (P1)

<a id="heading-10"></a>
### 2.1. Refresh token не ротируется
- **Файл:** `src/app/api/auth/refresh/route.ts`
- **Проблема:** При обновлении выдаётся новый access token, но refresh token остаётся тем же.
- **Исправление:** Ротация refresh token при каждом использовании.

<a id="heading-11"></a>
### 2.2. SQLite для production
- **Файл:** `prisma/schema.prisma:7`
- **Проблема:** `provider = "sqlite"` — нет concurrent writes, нет row-level locking, limited full-text search.
- **Исправление:** Миграция на PostgreSQL для production.

<a id="heading-12"></a>
### 2.3. Статусы как строки без enum
- **Файл:** `prisma/schema.prisma` (все модели)
- **Проблема:** `status String @default("draft")` — нет защиты на уровне БД от невалидных статусов.
- **Исправление:** Использовать Prisma enums для ролей и статусов.

<a id="heading-13"></a>
### 2.4. Transfer не перемещает между складами
- **Файл:** `src/app/api/inventory-movements/route.ts:56`
- **Проблема:** Операция `transfer` вычитает количество, но не создаёт запись на пункте назначения.
- **Исправление:** Транзакция: вычитание из source + прибавление в destination.

<a id="heading-14"></a>
### 2.5. Нет автосписания со склада при начале производства
- **Файл:** `src/app/api/production-orders/[id]/status/route.ts`
- **Проблема:** Смена статуса заказа на `in_progress` не уменьшает `StorageItem.quantity`.
- **Исправление:** Автоматическое списание материалов при начале производства.

## 3. СРЕДНИЕ ПРОБЛЕМЫ (P2)

<a id="heading-16"></a>
### 3.1. Дублирование типов ApiResponse
- **Файлы:** `src/lib/api-response.ts` + `src/types/index.ts`
- **Проблема:** Определения `ApiResponse<T>` и `PaginatedData<T>` продублированы.
- **Исправление:** Оставить только в `src/types/index.ts`.

<a id="heading-17"></a>
### 3.2. Три копии formatCurrency
- **Файлы:** `utils.ts`, `pdf/index.ts`, `proposal-block-builder.ts`
- **Проблема:** Три разные реализации форматирования валюты.
- **Исправление:** Единая функция в `utils.ts`, переиспользовать.

<a id="heading-18"></a>
### 3.3. Boilerplate в API routes
- **Файлы:** 30+ файлов
- **Проблема:** Повторяющийся try/catch с requireAuth() в каждом роуте.
- **Исправление:** HOF-обёртка или middleware.

<a id="heading-19"></a>
### 3.4. CrudPage модалка без a11y
- **Файл:** `src/components/crud-page.tsx:283`
- **Проблема:** Собственная реализация модалки вместо Dialog. Нет focus trap, escape key, aria-labels.
- **Исправление:** Использовать компонент Dialog.

<a id="heading-20"></a>
### 3.5. Две таблицы не переиспользуются
- **Файлы:** `crud-page.tsx` + `ui/table.tsx`
- **Проблема:** CrudPage реализует свою таблицу, не используя Table компонент.
- **Исправление:** CrudPage должен использовать Table.

<a id="heading-21"></a>
### 3.6. Dashboard 8 fetch-запросов
- **Файл:** `src/app/(dashboard)/dashboard/page.tsx:48`
- **Проблема:** Весь анализ данных на клиенте, нет серверного aggregation endpoint.
- **Исправление:** Создать `/api/dashboard/stats` endpoint.

<a id="heading-22"></a>
### 3.7. Пустой next.config
- **Файл:** `next.config.ts`
- **Проблема:** Нет настроек `serverExternalPackages` для `better-sqlite3`, нет security headers.
- **Исправление:** Добавить необходимые конфигурации.

## 5. АРХИТЕКТУРНЫЕ РЕКОМЕНДАЦИИ

### 5.1. Ввести API middleware
```typescript
// src/lib/api-wrapper.ts
export function withAuth(handler: Function, roles?: string[]) {
  return async (req, params) => {
    const user = await requireAuth();
    if (roles && !roles.includes(user.role)) {
      return apiError('Доступ запрещён', 403);
    }
    return handler(req, params, user);
  };
}
```

export async function GET() {
  const [proposals, orders, tasks] = await Promise.all([
    prisma.proposal.groupBy({ by: ['status'], _count: true }),

<a id="heading-34"></a>
### 5.5. Server Components вместо Client для списков
Многие страницы списков (`/products`, `/contracts`, `/proposals`) используют `'use client'` и fetch на клиенте. Следует использовать Server Components с `async` для первичной загрузки данных.

## ЗАКЛЮЧЕНИЕ

Проект kppdf-5.0 — рабочий CRM/ERP с покрытием основных бизнес-процессов. Однако **безопасность** и **качество кода** требуют серьёзной работы. Критичные проблемы (JWT, seed, upload, валидация) должны быть исправлены до деплоя в продакшен. Архитектурные улучшения (middleware, Zod, Server Components) сделают код поддерживаемым и масштабируемым.
