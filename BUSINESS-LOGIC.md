## МИССИЯ ПРИЛОЖЕНИЯ

> **Приложение ПОМОГАЕТ человеку быстро делать свои задачи.**
> Оно подсказывает: на каком этапе он остановился, что делать дальше.
> Не превращается в рутину заполнения ненужных данных.
> Максимальная автоматизация. Минимум лишнего. Каждое действие приближает товар к реализации.

## 2. КЛЮЧЕВЫЕ ПРИНЦИПЫ СИСТЕМЫ

### 2.1 Автоматизация
- Всё что может быть посчитано или подставлено — делается автоматически
- Пользователь только подтверждает или корректирует
- Номера документов генерируются системой
- Данные не дублируются — передаются по цепочке

### 2.2 Прозрачность
- Каждый этап сохраняет историю: кто, когда, что сделал
- Открытый диалог (слайд-панель) вместо увода на другую страницу
- Документы не теряются — каждый хранится в своей таблице

### 2.3 Помощь, а не рутина
- Приложение подсказывает: «Вы остановились на этапе X. Следующий шаг — Y»
- Уведомления/индикаторы о необходимых действиях
- Чек-боксы: что сделано, что ждёт

### 2.4 Гибкость
- Возможность редактирования авто-заполненных полей (где это допустимо)
- Проверка дубликатов номеров
- API готово к подключению внешних систем (Inventor, AutoCAD, SolidWorks)

## 3. МОДЕЛЬ ТОВАРА — РАСШИРЕННАЯ (будущее)

```
Product (товар)
  │
  ├── Базовая информация: SKU, название, категория, цена, ед.изм.
  ├── Характеристики: вес, габариты, материал
  ├── Документы: паспорт качества, чертёж, сертификаты
  ├── Фотографии
  │
  └── Модули/Изделия (ProductModule) ← БУДУЩЕЕ
        │
        ├── Название модуля
        ├── Артикул модуля
        ├── Габариты, вес
        ├── Чертежи модуля
        │
        └── Виды работ (ModuleWorkType)
              ├── Тип работы (сварка, столярка, сборка, покраска...)
              ├── Расчётное время (часов/дней)
              ├── Рабочий центр
              └── Материалы модуля (ModuleMaterial)
                    ├── Что нужно закупить
                    ├── Количество
                    └── Статус наличия на складе
```

**Принцип:** Конструктор дополняет товар модулями на этапе проектирования.
Один раз спроектировал → используется во всех последующих заказах.
Работа не дублируется.

## 4. ЭТАПЫ ДЕТАЛЬНО

### Этап 1–2: Товары → Витрина КП
- ✅ SKU авто-генерация: `[префикс_категории][4_цифры]`
- ✅ Типы: purchased (закупаемой) / manufactured (производимый)
- ✅ Цена + наценка по умолчанию
- ✅ Витрина: поиск, фильтр, корзина, A4 preview

### Этап 3: Коммерческое предложение (КП)
- ✅ Авто-номер `КП-XXXX` через `nextProposalNumber()`
- ✅ Снэпшот цен (фиксация на момент создания)
- ✅ Привязка к организации и клиенту
- ✅ Выбор шаблона документа
- ✅ Статусы: draft → sent → approved/rejected/converted
- ✅ PDF-генерация и скачивание
- ✅ **Поле RAL** (цвет покраски) — обязательное при создании КП
- 🔴 **История версий КП** — при переделке сохранять предыдущие версии

### Этап 4: Договор
- ✅ Авто-номер `Д-XXXX`
- ✅ Конвертация из КП: `POST /api/proposals/[id]/convert`
- ✅ Перенос товаров, статус КП → converted

### Этап 5: Оплата → Производство (КРИТИЧЕСКИЙ РАЗРЫВ)
- 🔴 **Авто-конвертация:** при смене статуса КП на «оплачено» → автоматически создаётся ProductionOrder
- 🔴 Номер заказа = номер КП (с возможностью редактирования)
- 🔴 Проверка дубликатов номеров заказов (предупреждение, блокировка)
- 🔴 В заказ попадает всё необходимое для производства (без цен)
- 🔴 Поля `contractId`, `proposalId` в ProductionOrder

### Этап 6: Производственный заказ
- 🔴 Авто-номер `ЗК-XXXX` через `nextProductionOrderNumber()`
- 🔴 Привязка к Contract и Proposal
- 🔴 Список товаров с их модулями и работами
- 🔴 Статусы заказа: created → designing → supplying → manufacturing → painting → shipping → completed

### Этап 7: Гантт (диаграмма планирования)
- ✅ Визуализация заказов и задач на временной шкале
- ✅ Day/Week/Month пресеты зума
- ✅ Today-линия + кнопка «Сегодня»
- ✅ Plan vs Actual (плановая и фактическая полосы)
- ✅ Прогресс-бары
- ✅ Фильтры по статусу, типу, сотруднику
- ✅ Панель загрузки сотрудников
- ✅ Выходные (сб/вс) — затенённые колонки
- ✅ Клик → слайд-панель с деталями (не уводит со страницы!)
- 🔴 **DnD-редактирование:** перетаскивание задач мышью для изменения сроков
- 🔴 Авто-заполнение из модулей товара (когда конструктор добавит работы и сроки)
- 🔴 Цвета по приоритету заказа

### Этап 8: Снабжение (НЕТ МОДУЛЯ)
- 🔴 Авто-формирование списка закупок из модулей товара
- 🔴 Статусы: требуется → заказано → получено
- 🔴 Уведомление снабженцу: «Для заказа ЗК-XXXX требуется закупить: ...»
- 🔴 Интеграция со складом (остатки)

### Этап 9: Изготовление
- 🔴 Рабочие видят свои задачи (только свои!)
- 🔴 Отметка о выполнении этапа
- 🔴 Учёт времени (план vs факт)
- 🔴 Переход к следующему этапу

### Этап 10: Покраска
- 🔴 Поле RAL из КП → обязательно видно на этом этапе
- 🔴 Если RAL не указан — пометка «Согласовать с заказчиком»
- 🔴 Статус покраски: ожидает → в работе → готово

### Этап 11: Отгрузка (НЕТ МОДУЛЯ)
- 🔴 Акт приёма-передачи (авто-заполнение из заказа)
- 🔴 Частичная отгрузка (можно отгрузить часть, остальное потом)
- 🔴 Фото отгруженного товара
- 🔴 История: что, когда, куда отгрузили

### Этап 12: Закрытие заказа
- ⚠️ OrderClosing CRUD есть
- 🔴 Авто-номер `ЗР-XXXX`
- 🔴 Авто-заполнение из ProductionOrder
- 🔴 Акт закрытия работ
- 🔴 Финансовый учёт: затраты vs выручка

## 5. РОЛЕВАЯ МОДЕЛЬ (будущее)

| Роль | Видит | Может |
|------|-------|-------|
| **Администратор** | Всё | Управление пользователями, ролями, настройки |
| **Менеджер по продажам** | Товары, витрина, КП, клиенты | Создание КП, отправка, согласование |
| **Конструктор** | Товары, модули, чертежи | Дополнение товара модулями и работами |
| **Снабженец** | Склад, закупки | Управление закупками, остатками |
| **Начальник производства** | Все заказы, Гантт | Распределение работ, контроль сроков |
| **Рабочий** | Свои задачи | Отметка выполнения, просмотр чертежей |
| **Бухгалтер** | Финансы, счета, акты | Закрытие заказов, сверки |

**Принцип:** Каждый видит ТОЛЬКО свои панели. Остальное скрыто.
При назначении роли — интуитивно понятные категории с подсказками.

## 10. ПРИНЦИПЫ РАБОТЫ С ДАННЫМИ

### Сохранение
- Черновики: localStorage каждые 2 сек
- Подтверждённые данные: сразу в БД
- История версий: при переделке КП — сохранять предыдущую версию

### Уведомления
- Приложение НЕ отправляет email/SMS само
- Индикаторы в интерфейсе: «Требуется действие»
- Пользователь сам скачивает PDF и отправляет клиенту

### Безопасность
- Номера документов: проверка уникальности
- Редактирование номеров: разрешено, но с проверкой дубликатов
- Роли: каждая роль видит только свои данные

### Масштабирование
- API готово к внешним интеграциям (REST)
- Импорт/экспорт через стандартные эндпоинты
- Модульная архитектура: каждый этап — отдельный модуль
