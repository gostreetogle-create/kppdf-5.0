**Файл:** `prisma/schema.prisma`
**Что:** Добавить `contractId String?` и `proposalId String?` в модель `ProductionOrder`. Добавить обратные связи: `Contract.productionOrders[]` и `Proposal.productionOrders[]`. Добавить индексы на оба поля.
**Зачем:** Производственный заказ должен знать из какого договора и КП он создан. Это критично для трассировки: «откуда пришёл заказ». Без этой связи цепочка КП→Договор→Производство разорвана.
**Проверка:** Prisma migrate проходит без ошибок. Можно создать ProductionOrder с contractId и proposalId. Можно получить Contract и увидеть его productionOrders.

### 1.3 ✅ API POST /api/contracts/[id]/convert-to-production
**Файл:** `src/app/api/contracts/[id]/convert-to-production/route.ts`
**Что:** Создать API-эндпоинт который:
1. Получает договор с товарами (contract.items) и КП (proposal.items с product)
2. Проверяет что нет уже созданного ProductionOrder для этого договора
3. Генерирует номер `ЗК-XXXX` через `nextProductionOrderNumber()`
4. Создаёт ProductionOrder с задачами (по одной на каждый товар) в транзакции
5. Обновляет статус договора на 'active'
6. Связывает contractId и proposalId
**Зачем:** Это мост между продажами и производством. Менеджер заключает договор → нажимает «Передать в производство» → система создаёт заказ с задачами. Без этого API каждый заказ создаётся вручную с дублированием данных.
**Проверка:** POST с валидным contractId → ProductionOrder создан, статус договора 'active', задачи созданы по товарам. Повторный POST → ошибка «уже существует».

**Проверка:** Открыть договор в статусе draft с товарами → кнопка видна. Нажать → лоадер → success-баннер. Статус договора локально обновлён на 'active'. Открыть тот же договор → кнопка скрыта.

### 1.5 🔴 Поле workCenterId в OrderTask

**Что:** Добавить `workCenterId String?` и связь `workCenter WorkCenter?` в модель `OrderTask`. Добавить индекс.
**Зачем:** Задача производства должна быть привязана к конкретному рабочему центру (цеху). Сейчас задача — просто текст без привязки к месту выполнения. Это делает Гантт и распределение работ бесполезными — непонятно КТО и ГДЕ делает задачу.

### 1.6 🔴 Авто-конвертация КП→ProductionOrder при статусе «оплачено»
**Файл:** `src/app/api/proposals/[id]/route.ts` (PATCH)
**Что:** При изменении статуса КП на 'paid' (оплачено) автоматически вызывать логику создания ProductionOrder:
1. Создать ProductionOrder из proposal.items (товары КП)
2. Номер заказа = номер КП по умолчанию (с возможностью редактирования)

**Зачем:** Ключевой принцип системы: оплата = автоматический запуск производства. Менеджер не должен вручную переносить данные. Это устраняет человеческий фактор (забыли создать заказ) и дублирование работы.
**Проверка:** Поменять статус КП на «оплачено» через PATCH → ProductionOrder создан автоматически. Номер заказа = номер КП. Товары перенесены как задачи.

## 🟡 ФАЗА 2: АВТО-НОМЕРА + БЛОКИРОВКИ (устранение ручного ввода)

### 2.1 ✅ Авто-номера в формах (7 форм)
**Файлы:**
- `src/app/(dashboard)/production/page.tsx` — `nextProductionOrderNumber()` → `ЗК-XXXX`
- `src/app/(dashboard)/warehouse/purchases/page.tsx` — `nextSupplierOrderNumber()` → `ЗП-XXXX`
- `src/app/(dashboard)/admin/tenders/page.tsx` — `nextTenderNumber()` → `Т-XXXX`
- `src/app/(dashboard)/admin/certificates/page.tsx` — `nextCertificateNumber()` → `С-XXXX`
- `src/app/(dashboard)/admin/rpp-entries/page.tsx` — `nextRppNumber()` → `РПП-XXXX`
- `src/app/(dashboard)/finance/order-closings/page.tsx` — `nextOrderClosingNumber()` → `ЗР-XXXX`
- `src/app/(dashboard)/finance/reconciliation/page.tsx` — `nextReconciliationNumber()` → `АС-XXXX`

**Что:** В каждой форме при создании новой записи номер поля генерируется автоматически через `useEffect` или `openCreate()`. Поле становится `readOnly` при редактировании существующей записи. Placeholder показывает «Авто-генерация...».
**Зачем:** Пользователь не должен вручную придумывать номера документов. Это приводит к ошибкам (пропуски, дубликаты, разные форматы). Система гарантирует уникальность и единый формат.
**Проверка:** Открыть каждую форму → при создании номер заполнен автоматически. При редактировании — поле заблокировано. Формат номера соответствует префиксу.

### 2.2 ✅ Блокировка номеров в ProposalForm и ContractForm
**Файлы:** `src/app/(dashboard)/proposals/page.tsx`, `src/app/(dashboard)/contracts/page.tsx`
**Что:** При редактировании КП или договора поле «Номер» становится `readOnly`. При создании — placeholder «КП-XXXX (авто)» / «Д-XXXX (авто)», номер генерируется API.
**Зачем:** Номер документа — это его идентификатор в системе. Изменение номера существующего документа ломает ссылки из других документов. Номер меняется ТОЛЬКО при создании.
**Проверка:** Открыть КП на редактирование → поле «Номер» заблокировано. Создать новый КП → номер генерируется API, поле доступно пока не сохранено.

### 2.3 🔴 Проверка дубликатов номеров
**Файлы:** Все API роуты где есть поле number
**Что:** При создании/обновлении документа проверять `prisma.model.findFirst({ where: { number } })`. Если дубликат → возвращать ошибку 400 с понятным сообщением «Документ с номером X уже существует».
**Зачем:** Даже если номер генерируется автоматически, пользователь может его отредактировать. Система должна гарантировать уникальность.
**Проверка:** Создать КП с номером «КП-0001». Попытаться создать второй с тем же номером → ошибка 400.

## 🟠 ФАЗА 3: МОДУЛИ ТОВАРА + СНАБЖЕНИЕ (расширение модели товара)

### 3.1 🔴 ProductModule — модули/изделия товара
**Файл:** `prisma/schema.prisma` (новая модель)
**Что:** Создать модель `ProductModule`:
- `id`, `productId` (→ Product), `name`, `article`, `width/height/depth`, `weight`, `image`, `createdAt`
- Связь: Product.modules → ProductModule[]
**Зачем:** Товар состоит из модулей/изделий. Это основа производственного планирования. Без модулей нельзя рассчитать сроки и загрузку — непонятно из чего состоит товар и сколько времени займёт его изготовление.
**Бизнес-контекст:** Конструктор на этапе проектирования разбирает товар на модули. Например, стол = столешница + ножки + крепёж. Каждый модуль имеет свои габариты и вес. Эта работа делается ОДИН раз и используется во всех последующих заказах с этим товаром.
**Проверка:** Создать товар → добавить к нему 2 модуля → модули сохраняются. Получить товар → модули включены в ответ.

### 3.2 🔴 ModuleWorkType — виды работ модуля
**Файл:** `prisma/schema.prisma` (новая модель)
**Что:** Создать модель `ModuleWorkType`:
- `id`, `moduleId` (→ ProductModule), `workTypeId` (→ WorkType), `estimatedHours`, `sortOrder`
- Каждый модуль может требовать несколько видов работ (сварка → столярка → покраска)
**Зачем:** Связывает модули с производственными работами. Определяет последовательность и время. Это то что автоматически заполняет Гантт и рассчитывает сроки.
**Бизнес-контекст:** Конструктор указывает: «для этого модуля нужна сварка (2 часа), потом шлифовка (1 час), потом покраска (3 часа)». Система суммирует и понимает общее время изготовления товара.
**Проверка:** Добавить модуль → указать 3 вида работ с временем → сохраняется. Общее время модуля = сумма часов.

### 3.3 🔴 ModuleMaterial — материалы модуля
**Файл:** `prisma/schema.prisma` (новая модель)
**Что:** Создать модель `ModuleMaterial`:
- `id`, `moduleId` (→ ProductModule), `name`, `quantity`, `unit`, `isPurchased` (закупается или есть на складе)
**Зачем:** Определяет что нужно для изготовления модуля. Автоматически формирует список закупок для снабжения.
**Бизнес-контекст:** Модуль «столешница» требует: ДСП 1200×600 (1 лист), кромка ПВХ (4 метра), клей (100 мл). Снабженец видит этот список и знает что закупать. Если материал есть на складе — не закупается.
**Проверка:** Добавить модуль → указать 2 материала → сохраняется. Статус материала: purchased (закупать) / in-stock (на складе).

### 3.4 🔴 Страница редактирования товара с модулями
**Файл:** `src/app/(dashboard)/products/[id]/page.tsx` (новая или расширенная)
**Что:** На детальной странице товара добавить вкладку «Модули и состав». Интерфейс: список модулей с возможностью добавить/редактировать/удалить. Для каждого модуля — подсписок видов работ и материалов.
**Дизайн:** Категорийный подход — всё разбито на логические блоки. Не перегружено. Интуитивно понятно.
**Зачем:** Конструктору нужно удобное место для работы с составом товара. Это не должно быть скрыто в админке или в отдельном разделе — это часть карточки товара.
**Проверка:** Открыть товар → вкладка «Модули» → добавить модуль → добавить работы и материалы → сохранить. Всё отображается структурированно.

### 3.5 🔴 Модуль снабжения: авто-список закупок
**Файл:** `src/app/(dashboard)/warehouse/purchases/page.tsx` (расширенная)
**Что:** При создании закупки дать возможность выбрать «Сформировать из заказа» → выбрать ProductionOrder → система собирает все ModuleMaterial с isPurchased=true из всех товаров заказа → создаёт список позиций закупки.
**Зачем:** Снабженец не должен вручную собирать что нужно купить. Система знает состав товаров и автоматически формирует список.
**Проверка:** Создать заказ с товарами у которых есть модули с материалами → нажать «Сформировать закупку из заказа» → список позиций заполнен автоматически.

### 3.6 🔴 Интеграция снабжения со складом
**Файл:** `src/app/api/storage-items/route.ts`, API модуля снабжения
**Что:** При отметке закупки как «получено» → автоматически обновлять складские остатки. При создании закупки проверять остатки на складе (isPurchased=false если материал есть).
**Зачем:** Склад и снабжение — две стороны одного процесса. Закупили → попало на склад. Есть на складе → не закупаем.
**Проверка:** Отметить закупку «получено» → складские позиции обновлены.

## 🔵 ФАЗА 4: ГАНТТ + ПРОИЗВОДСТВО (планирование и исполнение)

### 4.1 🔴 DnD-редактирование сроков на Гантте
**Файл:** `src/components/ui/gantt-chart.tsx`
**Что:** Добавить возможность перетаскивать задачи мышью для изменения сроков:
- Влево/вправо — сдвиг всей задачи по времени
- Левый/правый край — изменение даты начала/окончания (resize)
- При изменении → API PATCH на `/api/order-tasks/[id]` с новыми датами
- Визуальная обратная связь: тень, подсветка целевой даты, snapping к дням
**Зачем:** Гантт — инструмент ПЛАНИРОВАНИЯ, а не просто просмотра. Начальник производства должен мышкой перетащить задачу на другой день и система запомнит изменения. Текущая реализация — только чтение.
**Сложность:** Высокая. Требуется аккуратная работа с dnd-kit на временной шкале. Возможны проблемы с точностью позиционирования на разных зумах.
**Проверка:** Открыть Гантт → перетащить задачу на +2 дня → даты обновились в БД. Перезагрузить страницу → изменения сохранились.

### 4.2 🔴 Авто-заполнение Гантта из модулей товара
**Файл:** `src/app/api/order-tasks/route.ts`, логика при создании ProductionOrder
**Что:** При создании ProductionOrder (из КП или договора):
1. Для каждого товара найти его ProductModule[]
2. Для каждого модуля найти ModuleWorkType[]
3. Создать OrderTask на каждый вид работы с estimatedHours
4. Установить плановые даты: равномерно распределить общее время по доступным рабочим дням
5. Учитывать выходные (сб/вс)
6. Учитывать приоритет заказа
**Зачем:** После того как конструктор описал состав товара (модули + работы + время), система автоматически строит план производства. Начальнику остаётся только подтвердить или скорректировать.
**Проверка:** Создать товар с модулями и работами → создать заказ с этим товаром → Гантт автоматически заполнен задачами с датами.

### 4.3 🔴 Статусная модель производства
**Файл:** `prisma/schema.prisma` (поле status), `src/app/(dashboard)/production/page.tsx`
**Что:** Расширить статусы ProductionOrder и OrderTask:
- ProductionOrder: created → designing → supplying → manufacturing → painting → shipping → completed
- OrderTask: pending → in_progress → completed → blocked
- При переходе задачи в completed проверять: все ли задачи completed → перевести заказ на следующий этап
**Зачем:** Система должна отслеживать на каком этапе находится заказ. Каждый участник видит актуальный статус. Переходы между этапами частично автоматизированы (все задачи готовы → следующий этап).
**Проверка:** Создать заказ → статус 'created'. Завершить все задачи → заказ переходит в 'completed'.

### 4.4 🔴 Панель рабочего «Мои задачи»
**Файл:** `src/app/(dashboard)/production/workers/[id]/page.tsx` или новая страница
**Что:** Страница где рабочий видит ТОЛЬКО свои задачи:
- Список задач назначенных на него
- Сроки, описание, чертежи (если есть)
- Кнопка «Начать работу» / «Завершить»
- Учёт времени: план vs факт
- Фильтр по статусу задачи
**Зачем:** Каждый видит только свои панели. Рабочий не должен видеть весь заказ — только то что назначено ему. Это упрощает интерфейс и фокусирует на задачах.
**Проверка:** Назначить задачи на рабочего → открыть панель → видны только его задачи. Отметить выполнение → статус задачи обновлён.

3. При конвертации КП → заказ, RAL передаётся в ProductionOrder
4. На этапе покраски виден RAL-код и название цвета
5. Если RAL не указан — пометка «Согласовать с заказчиком»
**Бизнес-контекст:** Цвет покраски — обязательный параметр товара. Менеджер согласовывает RAL с клиентом на этапе КП. Код передаётся по всей цепочке. Если не согласован — система напоминает, но не блокирует.
**Проверка:** Создать КП → выбрать RAL → создать заказ → RAL виден на странице заказа.

### 5.4 🔴 Закрытие заказа: авто-заполнение + акты
**Файл:** `src/app/(dashboard)/finance/order-closings/page.tsx`, API
**Что:**
1. При создании закрытия → выбрать ProductionOrder → авто-заполнение:
   - Номер ЗР-XXXX (уже ✅)
   - Товары из заказа
   - Суммы (затраты из закупок, выручка из договора)
   - Даты из Гантта (план vs факт)
2. Финансовый расчёт: затраты vs выручка = маржа
3. Сохранение в БД с привязкой к заказу
**Бизнес-контекст:** Бухгалтер закрывает заказ: сравнивает плановые затраты с фактическими, выручку с расходами. Система автоматически подтягивает все цифры из связанных документов. Бухгалтер только проверяет и подтверждает.
**Проверка:** Создать закрытие из заказа → поля заполнены из заказа/договора/закупок. Отредактировать при необходимости → сохранить.
