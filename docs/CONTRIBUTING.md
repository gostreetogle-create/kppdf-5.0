# CONTRIBUTING — Правила для ИИ-агентов при работе с проектом kppdf-5.0

**Дата**: 2026-06-20
**Целевая аудитория**: все AI-агенты (Агент A / Агент B / будущие агенты), а также человек-разработчик.
**Назначение**: «Конституция» проекта. Соблюдение обязательно для любого изменения кодовой базы.

---

## ⚠️ ОСНОВНОЕ ПРАВИЛО: ничего не переписывать зря

Если модуль **работает** и **стабилен**, он **не подлежит рефакторингу**. Цикл проекта:

```
НОВЫЙ ФУНКЦИОНАЛ + БАГ-ФИКСЫ (расширение через новые экспорты, не переписывание)
```

**Запрещено**:
- «Улучшить» стабильный код «для красоты».
- Заменить одну имплементацию на «более правильную» без bug-репорта.
- Переименовать / переместить / переструктурировать стабильные модули.

---

## Правило 1. Проверять stability registry

Перед изменением **любого файла** агент обязан:

1. Открыть [`STABLE-MODULES.md`](./operations/STABLE-MODULES.md).
2. Проверить, не находится ли файл в списке **Tier A (стабильные)**.
3. Если файл в **Tier B (API frozen)** — соблюдать ограничения API (см. STABLE-MODULES.md).
4. Если файл **не в списке** — это Tier C/D/обычный код, можно менять свободно.

**Где искать**:
- `STABLE-MODULES.md` — в корне проекта.
- `docs/decisions/001-architecture-boundaries.md` — какие технологии зафиксированы.

---

## Правило 2. Если модуль стабилен — остановиться

Если агент планирует изменить модуль из Tier A, агент **обязан**:

1. **Прекратить** изменение немедленно.
2. **Сформулировать проблему**: что **не работает** в стабильном модуле? Это **реальный bug**, или «предпочтение эстетики»?
3. Если проблема **НЕ является реальным bug** — предложить альтернативу:
   - Расширить API (добавить новый export).
   - Создать новый модуль рядом (не трогая стабильный).
4. Если проблема **является реальным bug** — оформить **bug-fix patch** + ADR.

**Bug-fix patch** допустим, если:
- Есть конкретный repro (браузер + шаги).
- Поведение нарушает функциональные AC.
- Минимальный patch (НЕ cosmetic-only).

---

## Правило 3. Любое изменение стабильного модуля → ADR

Если изменение стабильного модуля **всё-таки необходимо** (даже bug-fix), агент обязан:

1. **Создать ADR** в `docs/decisions/ADR-NNN-{name}.md` со статусом `proposed`.
2. Включить в ADR:
   - Ссылку на bug-репорт или issue.
   - Описание текущего поведения vs желаемого.
   - Оценку стоимости изменения.
   - Альтернативы (минимум 2 — например, «расширить через новый export» vs «изменить существующий»).
3. **Получить `accepted`** от архитектора (Агент A или человек).
4. Только после `accepted` — менять код.

Пока ADR в статусе `proposed`, **никакого кода не пишется**.

---

## Правило 4. Новый функционал = расширение, а не переписывание

Когда нужно добавить новый функционал:

| ✅ Правильно | ❌ Неправильно |
|--------------|-----------------|
| Добавить новый export в `jwt.ts`: `verifyRefreshToken` | Изменить сигнатуру `verifyToken()` (breaking) |
| Создать новый `lib/business-rules/` для новой логики | Переписать `lib/auth.ts` «более чисто» |
| Расширить PDFGenerator API через opts | Изменить сигнатуру `generateProposalPdf` |
| Добавить новый route handler `POST /api/foo` | Изменить существующий `POST /api/foo` |

**Когда переписывание допустимо**:
- Существующая имплементация — outright **broken** (security, data corruption).
- Новая feature **невозможна** без breaking changes — нужно обосновать в ADR.

---

## Правило 5. Тесты обязательны

Каждый PR / cycle должен:

1. **Покрывать новый код тестами**:
   - Unit тесты для helpers (Vitest).
   - Route handler тесты для API (Vitest с mocked prisma).
   - Smoke тесты для UI (Playwright, если затрагивает UI).
2. **Не ломать существующие тесты**:
   - `npx tsc --noEmit` → 0 errors.
   - `npx vitest run` → все suites зелёные.
   - `npx eslint src --max-warnings=999` → 0 errors.
3. **Падение тестов — стоп-фактор**:
   - Цикл НЕ считается завершённым, пока тесты не зелёные.

Если тесты **невозможны** для какого-то кода (например, PDF generation), это должно быть:
- Явно зафиксировано в STABLE-MODULES.md (Tier B: API frozen, нет тестов).
- Запланировано в цикле 48-49 (Tests isolation / integration mock prisma).

## Правило 6. Commit discipline

Каждый цикл = **один тематический commit** (или несколько логически связанных):

```bash
# Пример хорошего сообщения:
git commit -m "cycle-51: B.1 — auto-create InventoryMovement IN при ProductionOrder completed"

# Пример плохого сообщения:
git commit -m "fix"
git commit -m "wip"
git commit -m "various changes"
```

Если цикл требует нескольких commits (например, cycle 42-43 versioning) — допустимы commits с суффиксом `-part-N`:
```bash
git commit -m "cycle-42-43: Block 3.2 — Proposal versioning, part 1 (schema migration)"
git commit -m "cycle-42-43: Block 3.2 — Proposal versioning, part 2 (helper + API + UI)"
```

---

## Правило 7. Документация обновляется вместе с кодом

Каждый цикл обновляет **все** следующие документы:

| Документ | Что обновляется |
|----------|-----------------|
| `tasks/current-task.md` | Раздел `=== РЕЗУЛЬТАТ ===` в конце цикла |
| `audit-log.md` | Запись цикла (формат см. предыдущие записи) |
| `audit-tasks.md` | Статус блока (`📋 planned` → `✅ DONE`) + commit хэш |
| `audit-tasks-business.md` | Если блок относится к бизнес-плану |
| `STABLE-MODULES.md` | Если модуль перешёл в Tier A или Tier B |
| `docs/decisions/ADR-NNN.md` | Если принято архитектурное решение |
| `CURRENT-CHECKLIST.md` | Обновляется ПЕРЕД началом и ПОСЛЕ завершения каждого цикла |

Нет «забытых» документов. Если документ не обновлён — цикл не завершён.

## Workflow для типовых ситуаций

### Ситуация A: добавление нового feature (например, новый модуль)

```
1. Прочитать STABLE-MODULES.md и ADR-001 (architecture boundaries).
2. Проверить, не задевает ли это Tier A или стабильные API.
3. Спланировать cycle (tasks/current-task.md).
4. Имплементировать + тесты.
5. Gates: tsc 0, vitest green, eslint 0.
6. Code-reviewer pass.
7. Обновить ВСЕ документы (см. Правило 7).
8. Один commit.
9. Если новый модуль достиг Tier A/B → обновить STABLE-MODULES.md.
```

### Ситуация B: bug-fix в стабильном модуле

```
1. Прочитать STABLE-MODULES.md.
2. Оформить bug-issue (root cause).
3. Создать ADR-NNN (proposed → accepted).
4. Минимальный патч.
5. Тесты для патча + проверка, что существующие тесты зелёные.
6. Gates same.
7. Code-reviewer pass.
8. Обновить документы.
9. Один commit.
```

**Запрещено**: «сразу фиксить без ADR» — даже если фикс тривиальный.

### Ситуация C: рефакторинг по запросу «сделай чище»

```
1. СТОП. Не делать.
2. Оформить ADR-NNN (proposed).
3. В ADR указать:
   - Конкретное улучшение (НЕ «для чистоты», а «уменьшит X на Y%»).
   - Альтернативу: расширить через новый export или новый модуль.
   - Стоимость vs выгода.
4. Если ADR `accepted` — выполнить как обычный цикл.
5. Если ADR `rejected` / `superseded` — не делать.
```

### Ситуация D: меняется стек (`давай перейдём на X`)

```
1. СТОП. Прочитать ADR-001.
2. Создать ADR-002+ (proposed) с обоснованием.
3. Включить в ADR:
   - Конкретная проблема с текущей технологией (1+ конкретный bug/perf issue).
   - 2+ альтернативы.
   - Стоимость миграции.
   - План rollback.
4. Получить `accepted` от архитектора.
5. Только тогда — цикл на миграцию.
```

**Запрещено**: менять стек без ADR-002+. См. ADR-001.

## Роли

| Роль | Кто | Обязанности |
|------|-----|-------------|
| **Агент A** (Критик-Архитектор) | Главный контролёр качества | Ревью ADR, одобрение изменений стабильных модулей, приоритизация циклов |
| **Агент B** (Рецензент-Исполнитель) | Имплементатор + валидатор | Реализация циклов, поддержка тестов, репорты в audit-log.md |
| **Человек-разработчик** | Пользователь | Финальное одобрение для необычных изменений |

---

## Антипаттерны (явно запрещено)

| Антипаттерн | Почему плохо | Что делать вместо |
|-------------|--------------|-------------------|
| «Чисто рефакторинг» в cycle N | Без бизнес-причины | Оформить как ADR-NNN (proposed) |
| «Переписать на X» | Нарушает ADR-001 | Сначала ADR-002+ |
| «Это просто фикс одного бага» в Tier A | Должен идти через ADR | Минимальный патч + ADR |
| Большой PR с 10 файлами | Трудно ревью, высокий риск регрессии | Разбить на cycle-N + cycle-N+1 |
| Тесты помечены `.skip` или `.todo` | Не покрыты, могут регрессировать | Реальные тесты или явно задекларировано в STABLE-MODULES.md |
| Комментарий «TODO» в коде | Должен быть issue | Создать issue или cycle |

---

## Enforcement

Эти правила **не enforced** автоматически (нет pre-commit hook для проверки).
**Enforcement** происходит через:
- Code-review (Агент A).
- Человек-разработчик (финальное ревью).
- Conversation discipline: агенты **обязаны** ссылаться на этот файл при любых изменениях.

Если правила нарушаются: **исправить немедленно** (откатить, оформить ADR, продолжить по правилам).
