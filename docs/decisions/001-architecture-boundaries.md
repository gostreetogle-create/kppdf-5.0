# ADR-001: Architecture Boundaries — тех-стек kppdf-5.0 зафиксирован

**Дата**: 2026-06-20
**Статус**: accepted
**Заменяет**: (нет)

---

## Контекст

Проект kppdf-5.0 прошёл стадию прототипирования (cycles 0-43). Стек уже стабильно используется в production:
- **Frontend**: Next.js 16.2.9 (App Router) + React 19 + Tailwind CSS.
- **DB / ORM**: Prisma 7.8.0 + PostgreSQL (через PrismaPg driver adapter из `@prisma/adapter-pg`).
- **State**: Zustand (для auth-store, theme-store).
- **PDF**: jsPDF + jspdf-autotable (динамический import через `getPdfLibs()`).
- **UI interactions**: dnd-kit (для sortable-блоков в templates, Gantt), `@dnd-kit/sortable`.
- **Tests**: Vitest 4 (88+ unit-тестов), Playwright (e2e smoke).
- **Forms validation**: Zod schemas в `src/lib/validations/`.
- **Auth**: JWT в HttpOnly cookies, refresh rotation.
- **Build deploy**: Docker + nginx для Synology DSM (см. `deploy/`).

В течение прототипирования несколько раз возникали предложения «давай перепишем на X»:
- «Может попробуем Drizzle ORM вместо Prisma?»
- «Может переходить на `@react-pdf/renderer` для server-side рендера?»
- «Может вместо Zustand — Redux Toolkit?»

Каждое такое предложение **не было** принято по дефолту, потому что переключение стека = инвестиция в рефакторинг с риском регрессии. Однако формальной защиты от повторных предложений нет — каждый раз приходится пересогласовывать.

### Что фиксирует этот ADR

Этот ADR **фиксирует** текущий стек как **канонический** для проекта. Любое предложение сменить одну из технологий **обязано пройти через отдельный ADR** (см. ADR-002+ ниже), где будут рассмотрены:
1. Конкретная проблема с текущим стеком.
2. Конкретное преимущество альтернативы.
3. Стоимость миграции (в человеко-часах или циклах).
4. Риск регрессии.
5. Обратная совместимость.

Без прохождения этой процедуры предложения «переписать на X» **НЕ должны реализовываться** в циклах проекта.

## Решение

### 1. Зафиксированный стек

| Слой | Технология | Версия | Где используется | Notes |
|------|-----------|--------|------------------|-------|
| Frontend Framework | Next.js | 16.2.9 | все страницы и API | App Router |
| UI Library | React | 19 | все компоненты | |
| Styling | Tailwind CSS | 4 | все стили | CSS variables для тем |
| ORM | Prisma | 7.8.0 | вся работа с БД | + driver adapter, НЕ datasourceUrl |
| DB Adapter | `@prisma/adapter-pg` + PrismaPg | latest | `src/lib/db.ts` | Prisma 7 modern pattern |
| DB | PostgreSQL | 14+ | production | (альтернативно `@prisma/adapter-better-sqlite3` установлен, но НЕ используется) |
| State | Zustand | latest | `src/stores/*.ts` | (auth, theme) |
| PDF | jsPDF | 4.2.1 | `src/lib/pdf/index.ts` | + jspdf-autotable 5.0.8 |
| DnD | `@dnd-kit/core` + `@dnd-kit/sortable` | latest | templates, Gantt | |
| Schema validation | Zod | latest | `src/lib/validations/*.ts` | request/response schemas |
| Auth | JWT (HttpOnly cookies) | latest | `src/lib/jwt.ts` + `src/lib/auth.ts` | refresh rotation implemented |
| Unit tests | Vitest | 4 | `src/lib/__tests__/*.test.ts` | 88+ tests |
| E2E tests | Playwright | latest | `tests/e2e/login.spec.ts` | smoke |
| Build | TypeScript | 5.x | весь проект | strict mode |
| Deploy | Docker + nginx | latest | `deploy/` | Synology DSM target |

### 2. Процедура изменения стека

Любая попытка заменить **одну из технологий** на альтернативу **обязана**:

1. **Оформить как новый ADR** в `docs/decisions/ADR-NNN-{name}.md` со статусом `proposed`.
2. **Включить в ADR**:
   - Конкретная проблема с текущей технологией (1+ реальный bug, performance issue, или конкретное ограничение — НЕ «мне больше нравится X»).
   - Оценка стоимости миграции (человеко-часы или циклы).
   - Риск регрессии (что может сломаться).
   - Альтернативы, которые были рассмотрены (минимум 2).
3. **Получить статус `accepted`** от архитектора (Агента A).
4. **Только после `accepted`** — планирование нового цикла на миграцию.

Пока ADR в статусе `proposed`, **никакого кода не пишется**.

### 3. Когда этот ADR можно пересмотреть

- Если выявлена **критическая security vulnerability** в зафиксированной технологии.
- Если выявлена **критическая блокирующая проблема**, которая не решается в пределах стека.
- Если проект переходит в режим **greenfield** (полностью новая кодовая база).

Обычные «улучшения» и «мне больше нравится» НЕ являются основанием для пересмотра.

## Альтернативы

### Альтернатива A: «Открытый стек, любое предложение приветствуется»

Не зафиксировать стек; каждое предложение рассматривать ad-hoc.

**Плюсы**:
- Гибкость; можем быстро попробовать новую технологию.

**Почему отвергли**:
- Loss of focus: в прототипе и так было несколько таких «попробуем» без последующего внедрения.
- Постоянный «тех-shakeup» без реальной причины.
- Регрессия-риск: переключение библиотек (особенно ORM, PDF-engine) ломает edge cases.
- Когнитивная нагрузка: каждый агент должен «угадывать» стек.

### Альтернатива B: «Зафиксировать версии с точностью до patch, без альтернатив»

Залочить не только технологию, но и конкретные версии + любые patch-апдейты запрещены.

**Плюсы**:
- Нулевая drift в зависимостях.

**Почему отвергли**:
- Security patches для Next.js / Prisma невозможно применить быстро (нужен новый ADR для security update).
- Overkill: drift можно контролировать через `package-lock.json` + Dependabot.

### Альтернатива C: «Зафиксировать только критические слои (DB, PDF), UI layer свободный»

Зафиксировать только Prisma и jsPDF, остальное оставить гибким.

**Плюсы**:
- Возможность менять UI framework (Next.js → Remix) без отдельного ADR.

**Почему отвергли**:
- Next.js App Router специфичен (route handlers, server components). Переход на Remix = полная переписка роутинга — это уже потеря большей части работы.
- Чёткие архитектурные слои как раз есть смысл защищать все, не только DB/PDF.

## Последствия

### Позитивные

- **Архитектурная стабильность**: каждый агент знает стек и не предлагает «случайные» изменения.
- **Фокус на бизнес-логике**: усилия идут на cycles 44-58 (тех-план + бизнес-план), а не на «тех-shakeup».
- **Документированная граница**: ADR-002+ должен обосновать **почему** нужно отклониться от ADR-001.
- **Меньше регрессий**: код стабилизируется через cycles, а не через постоянные «улучшения».

### Негативные

- **Новая бюрократия**: каждый «тех-переключатель» требует ADR. Это замедляет ad-hoc изменения.
- **Возможная устаревание**: если через 2-3 года появится значительно лучшая технология, формальный процесс её принятия может быть длительным.

### Реверс-условия

Решение нужно пересмотреть, если:
- В одной из зафиксированных технологий выявлена **CVE с severity Critical** и нет backward-compatible fix.
- Зафиксированная технология **перестала сопровождаться** (EOL), и есть только experimental форк.
- Проект радикально меняет scope (например, из CRM в e-commerce платформу), и текущий стек **объективно не подходит** для нового scope.

В таких случаях: новый ADR с `proposed` → анализ → `accepted` → цикл на миграцию.
